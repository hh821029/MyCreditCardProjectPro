# 演算法複雜度與設計取捨 (Time-Space Complexity Trade-offs)

本文件旨在回顧演算法中**時間複雜度 (Time Complexity)**與**空間複雜度 (Space Complexity)**的理論基礎，並結合本信用卡 ETL 與回饋計算引擎專案（`MyCreditCardProjectPro`）的實際應用場景，探討在不同情境下如何進行架構設計上的取捨 (Trade-off)。

---

## 1. 核心概念回顧

在軟體工程中，我們很少能同時獲得「最快的速度」和「最省的記憶體」。這兩者通常是互相制衡的：

*   **時間複雜度 (Time Complexity)**：衡量演算法執行時間隨輸入資料量 $N$ 增長的速度。通常以漸進符號 Big-O 表示。
*   **空間複雜度 (Space Complexity)**：衡量演算法在運行過程中，臨時佔用記憶體空間隨輸入資料量 $N$ 增長的速度。
*   **以空間換時間 (Space-for-Time Trade-off)**：藉由耗費額外的記憶體（例如快取、哈希表、索引）來儲存預先計算的資料，進而降低後續查詢或計算的時間。
*   **以時間換空間 (Time-for-Space Trade-off)**：為了節省記憶體，每次需要資料時都重新計算或進行磁碟/資料庫 I/O，這會增加運算時間。

---

## 2. 本專案之實務應用場景與取捨分析

在 `MyCreditCardProjectPro` 專案中，以下三個核心模組的設計最能體現時間與空間複雜度的取捨：

### 核心場景 A：回饋引擎規則匹配 (Waterfall Engine)
在計算每筆交易的回饋時，我們必須將交易與規則庫中的多條規則進行比對。

#### 方案一：線性搜尋 (以時間換空間)
*   **做法**：對每一筆交易，都在記憶體中順序遍歷 `bridge_reward_rules.csv` 的所有規則，直到找到符合條件者或結束。
*   **複雜度**：
    *   **時間複雜度**：$O(N \times M)$。其中 $N$ 為交易筆數，$M$ 為回饋規則數。
    *   **空間複雜度**：$O(1)$ 額外空間（只需將規則載入為一個簡單的 List 進行迭代）。
*   **缺點**：當交易量達數萬筆、規則數達上百條時，雙重迴圈將導致運算時間呈指數級上升，影響效能。

#### 方案二：多級雜湊索引優化 (以空間換時間) — ★ 本專案推薦
*   **做法**：在計算開始前，將規則庫載入並預先建立多層雜湊表 (Nested Dict)：
    $$\text{card\_type} \longrightarrow \text{mobile\_payment} \longrightarrow \text{list of rules (sorted by priority)}$$
*   **複雜度**：
    *   **時間複雜度**：$O(N + M)$。建立索引需時 $O(M)$，後續每筆交易的匹配時間降低至接近常數級 $O(1)$ 或 $O(L)$（$L$ 為單一卡片與支付管道下的少數規則數）。
    *   **空間複雜度**：$O(M)$，需要額外記憶體來維護這個雜湊結構。
*   **取捨決策**：在信用卡記帳情境中，規則數量 $M$ 通常在數百條以內，記憶體佔用極小（數百 KB 到數 MB），但交易數量 $N$ 會隨著時間累積。因此，**採用方案二（多級雜湊索引）以極小的記憶體空間代價，換取極快的計算速度，是極為划算的交易。**

---

## 核心場景 B：商家名稱正規化 (Merchant Normalization & Cache)
商家名稱清洗常需要使用正規表示式 (Regex) 或模糊比對（如 Levenshtein Distance）來剔除分店資訊、英文後綴等。

#### 方案一：無狀態逐筆比對
*   **做法**：每讀入一筆交易的 `merchant` 欄位，就將其傳入 Refiner 執行一系列的 Regex 替換。
*   **複雜度**：
    *   **時間複雜度**：$O(N \times R)$，其中 $R$ 為 Regex 規則數量與比對長度的開銷。
    *   **空間複雜度**：$O(1)$，不需要在記憶體中保留狀態。
*   **缺點**：相同商家（例如「統一超商」）在帳單中會重覆出現上百次，每次都重跑一次 Regex 會產生大量重複的 CPU 運算。

#### 方案二：記憶化快取 (LRU Cache / Dictionary Cache) — ★ 本專案推薦
*   **做法**：在內存中建立一個快取表 `merchant_cache: dict[str, str]`。在執行 Regex 之前先查詢快取，若命中則直接返回結果；若未命中，則執行清洗後將 `原始名稱 -> 顯示名稱` 寫入快取。
*   **複雜度**：
    *   **時間複雜度**：$O(U \times R + N)$，其中 $U$ 為**不重複**的商家數量 ($U \ll N$)。對於重複出現的商家，查詢時間為 $O(1)$。
    *   **空間複雜度**：$O(U)$，記憶體中需儲存 $U$ 組對照關係。
*   **取捨決策**：因為使用者的消費商家重複率通常高達 70% ~ 90%，建立商家對照快取可以節省大部分的 CPU Regex 計算時間，且 $O(U)$ 的空間開銷在個人記帳規模下微乎其微。

## 核心場景 C：資料載入與處理策略 (Data Loading Strategy)
從 SQLite 資料庫中讀取交易資料並執行 ETL 的方式。

#### 方案一：全量批次載入 (Pandas Batch Processing)
*   **做法**：使用 `pd.read_sql` 一次性將整個交易事實表讀入記憶體，並在 DataFrame 中進行關聯與計算。
*   **複雜度**：
    *   **時間複雜度**：由於記憶體內運算（向量化操作）效率極高，處理時間最短。
    *   **空間複雜度**：$O(N)$。若交易資料量暴增，記憶體佔用會隨之線性增長。

#### 方案二：串流/分塊處理 (Chunking / Generator)
*   **做法**：使用 SQL 的 `LIMIT / OFFSET` 或 Python 生成器 (Generator) 以 `chunk_size=1000` 逐批讀入並處理。
*   **複雜度**：
    *   **時間複雜度**：因為頻繁的資料庫 I/O 與多次運算調用，總耗時較長。
    *   **空間複雜度**：$O(C)$，其中 $C$ 為分塊大小，空間複雜度為常數級 $O(1)$，與總交易量無關。

#### 🎯 本專案的取捨決策
*   **目前階段**：考量到個人記帳的交易總量（通常在數萬筆以下，資料量 < 50MB），記憶體並非系統瓶頸。為了加速處理流程與簡化程式碼複雜度，**我們優先採用全量載入 (方案一)**。
*   **未來擴充性**：若日後此系統轉為雲端多使用者服務，資料量達百萬級以上時，必須轉為**分塊處理 (方案二)** 以防主機記憶體溢出 (OOM)。

## 3. 架構設計決策矩陣

以下總結我們在設計 ETL 與回饋引擎時的評估維度：

| 評估維度 | 時間優先 (空間換時間) | 空間優先 (時間換空間) | 本專案實務選擇與理由 |
| :--- | :--- | :--- | :--- |
| **回饋計算匹配** | 建立多級 Hash 索引 ($O(N)$) | 線性遍歷規則 ($O(N \times M)$) | **時間優先**：規則數少而交易數多，Hash 記憶體開銷可忽略。 |
| **商家正規化** | 記憶化快取 ($O(1)$) | 每筆重新計算 Regex | **時間優先**：商家重複率高，快取效益極大。 |
| **資料載入** | 一次性載入記憶體 (Pandas) | 串流分塊載入 (Generator) | **時間優先**：個人資料量小，速度為優先考量。 |
| **歷史結帳日查詢** | 快取 `dim_billing_history` | 每次以 SQL 查詢資料庫 | **時間優先**：歷史結帳日為靜態維度資料，預先載入有助於批次計算。 |

---

## 4. 結論

在 `MyCreditCardProjectPro` 這種中小型規模、資料偏向密集計算但總量適中的系統中：
1.  **記憶體空間非常充裕**：個人的電腦或輕量伺服器記憶體（如 512MB 以上）遠大於我們數萬筆交易所需要的內存。
2.  **計算響應時間 (Latency) 影響體驗**：使用者在 Web 前端或 CLI 執行 ETL 時，希望能立即看到分析結果。

因此，本專案的架構設計策略應明確傾向於**「以空間換時間」**：
*   儘可能在記憶體中快取常用的維度表（如 `dim_cards.csv`、`dim_billing_history.csv`）。
*   建立索引結構加速規則匹配。
*   使用快取減少重複的字串清洗運算。
